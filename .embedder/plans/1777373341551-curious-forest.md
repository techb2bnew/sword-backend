# Sword ERP Backend -- Module-wise Development Plan

## Current State
- Single `app.js` file with 2 product routes (GET/POST)
- Flat structure, no route separation, no auth, no middleware

## Target Architecture

```
server/
├── index.js              # Entry point
├── db.js                 # PostgreSQL pool
├── .env                  # Environment variables
├── package.json
├── config/
│   └── db.sql            # Full database schema
├── middleware/
│   └── auth.js           # JWT authentication middleware
├── routes/
│   ├── auth.js           # Register, Login
│   ├── users.js          # User CRUD, role management
│   ├── products.js       # Product CRUD (moved from app.js)
│   ├── customers.js      # Customer CRUD
│   ├── suppliers.js      # Supplier CRUD
│   ├── inventory.js      # Stock tracking, adjustments
│   ├── sales.js          # Invoices, invoice items
│   ├── purchases.js      # Purchase orders, purchase items
│   └── reports.js        # Sales/inventory/profit reports
└── app.js                # Route aggregator (imports all routes)
```

## Module Development Order

### Module 1: Database Schema + Project Restructure
- Create `config/db.sql` with all tables (users, products, customers, suppliers, invoices, invoice_items, purchases, purchase_items, inventory)
- Restructure project into `routes/` and `middleware/` directories
- Move existing product routes to `routes/products.js`
- Update `app.js` to aggregate all route files
- Install new dependencies: `bcryptjs`, `jsonwebtoken`

### Module 2: Users & Auth
- `routes/auth.js` -- POST `/api/auth/register`, POST `/api/auth/login`
- `routes/users.js` -- GET/PUT/DELETE users (admin only)
- `middleware/auth.js` -- JWT token verification, role check helper
- Password hashing with bcryptjs, JWT tokens

### Module 3: Product Management (enhance existing)
- `routes/products.js` -- Add PUT (update) and DELETE endpoints
- Add `category` and `stock_quantity` fields
- Protect routes with auth middleware

### Module 4: Customer Management
- `routes/customers.js` -- Full CRUD: GET all, GET by id, POST, PUT, DELETE
- Fields: name, email, phone, address

### Module 5: Supplier Management
- `routes/suppliers.js` -- Full CRUD
- Fields: name, contact_person, email, phone, address

### Module 6: Inventory Management
- `routes/inventory.js` -- GET stock levels, POST stock-in, POST stock-out
- Links to products table, tracks warehouse location and quantity changes

### Module 7: Sales / Billing
- `routes/sales.js` -- Create invoice, get invoices, get invoice by id
- Auto-calculate subtotal, discount, GST, grand total
- Create invoice items linked to products
- Deduct inventory on sale

### Module 8: Purchase Management
- `routes/purchases.js` -- Create purchase order, list, get by id
- Link to suppliers, create purchase items
- Add inventory on purchase completion

### Module 9: Reports & Analytics
- `routes/reports.js` -- GET endpoints for:
  - Sales summary (daily/monthly/total)
  - Inventory status (low stock alerts)
  - Profit/loss by date range
  - Top selling products

## API Prefix Convention
All routes will use `/api/` prefix:
- `/api/auth/login`, `/api/auth/register`
- `/api/products`, `/api/customers`, `/api/suppliers`
- `/api/inventory`, `/api/sales`, `/api/purchases`
- `/api/reports/sales`, `/api/reports/inventory`

## Key Design Decisions
- JWT-based auth with role-based access (admin, staff, viewer)
- All write operations protected by auth middleware
- Inventory auto-updated on sales and purchases
- GST calculated at invoice level
- PostgreSQL with parameterized queries (no ORM, keeping it lightweight)
